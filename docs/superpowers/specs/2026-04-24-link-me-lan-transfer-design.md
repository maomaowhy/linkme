# Link Me 局域网多端互传设计

## 1. 项目目标

构建一个从零开始的跨端互传项目，覆盖：
- 桌面端：Windows `.exe` / macOS `.dmg`，基于 `Electron`
- 移动端：Android `.apk`，基于 `Vue 3 + Vite + uni-app`

首期产品目标：
- 电脑与手机之间可互传文本、文件、多文件、文件夹
- 手机与手机之间可互传文本、文件、多文件、文件夹
- 同一台服务端可同时连接多个手机
- 建立连接时必须经由接收端人工允许
- 接收端支持主动断开连接
- 发送与接收时都支持用户选择保存位置

非目标（首期不做）：
- 公网穿透
- iOS 端
- 蓝牙主链路传输
- 自动后台长期保活
- 断点续传/暂停继续

## 2. 结论与技术选型

### 2.1 通信方式选择

本项目首期统一采用 **局域网 Wi‑Fi** 作为主链路，不采用蓝牙作为主传输方式。

原因：
- 文件和文件夹传输对吞吐要求高，蓝牙不适合大文件和多文件场景
- 多设备并发连接下，基于局域网的 `HTTP + WebSocket` 方案实现更稳定
- `Electron` 与 Android/uni-app 更容易围绕本地网络服务构建统一协议
- 手机与电脑、手机与手机都可以复用一套协议和状态管理模型

蓝牙在后续如需引入，仅作为“附近发现设备”的增强能力，不参与主数据传输。

### 2.2 总体方案

采用 **统一协议 + 多角色设备模型**：
- 任何设备都可以进入“接收模式”，成为临时服务端（Host）
- 其他设备扫码后作为客户端（Client）连接
- 控制信令走 `WebSocket`
- 文件与文件夹数据走 `HTTP`

这样可以统一支持：
- 电脑 ↔ 手机
- 手机 ↔ 手机
- 一台 Host ↔ 多台 Client

## 3. 分阶段实施策略

为控制实现风险，按两个阶段落地，但协议和目录结构一次设计到位。

### Phase 1
- 桌面端支持 Host 模式
- 手机端支持 Client 模式
- 打通电脑 ↔ 手机的文本、文件、多文件、文件夹互传
- 支持多手机同时连接同一台电脑

### Phase 2
- 手机端新增 Host 模式（接收模式）
- 打通手机 ↔ 手机互传
- 复用 Phase 1 的协议、任务模型和 UI 结构

实现时可以先把协议、共享类型、服务管理模块按完整模型设计好，再优先完成 Phase 1 的运行闭环。

## 4. 项目结构设计

采用 monorepo：

```text
link-me/
  apps/
    desktop/
    mobile/
  packages/
    shared/
  docs/
    superpowers/
      specs/
      plans/
```

### 4.1 `apps/desktop`
职责：
- `Electron main process` 启动本地服务
- 提供文件系统访问、目录选择、端口监听、本机 IP 获取
- 渲染层负责二维码展示、设备审批、会话列表、传输任务面板
- 打包为 Windows 与 macOS 桌面应用

### 4.2 `apps/mobile`
职责：
- `uni-app` Android 应用
- 提供扫码、连接、聊天、发送文件、接收文件、选择保存位置等界面
- Phase 2 中增加 Host 模式

### 4.3 `packages/shared`
职责：
- 统一协议消息定义
- 共享 TypeScript 类型
- 任务状态枚举
- 通用工具函数（路径、大小、二维码 payload、状态转换等）

## 5. 设备角色模型

### 5.1 Host
Host 是当前“接收模式”开启的设备，负责：
- 启动本地 `HTTP + WebSocket` 服务
- 显示二维码
- 接收连接申请
- 人工允许或拒绝连接
- 管理在线设备列表
- 接收或发送文本、文件、文件夹
- 主动断开某个设备

### 5.2 Client
Client 是扫码连接 Host 的设备，负责：
- 扫码解析连接信息
- 发起握手请求
- 等待 Host 审批
- 发送文本、文件、文件夹
- 接收 Host 推送的文本、文件、文件夹

### 5.3 会话模型
每个设备之间建立独立会话，核心字段包括：
- `sessionId`
- `localDeviceId`
- `remoteDeviceId`
- `role`
- `status`：`pending | connected | disconnected | rejected`
- `connectedAt`

同一 Host 可以维护多个 `sessionId`，彼此独立。

## 6. 连接建立流程

### 6.1 二维码内容
二维码只包含建立连接所需的最小元数据：
- `version`
- `hostId`
- `hostName`
- `hostIp`
- `port`
- `pairToken`（一次性或短时有效）
- `expiresAt`

### 6.2 握手流程
1. Host 启动本地服务并生成二维码
2. Client 扫码后校验 payload 格式和有效期
3. Client 发起 `WebSocket` 握手，提交自身设备信息与 `pairToken`
4. Host 将申请显示为“待允许设备”
5. 用户点击允许后，Host 下发 `sessionId`
6. 双方状态进入 `connected`
7. 任一方断开或被踢出后，会话状态进入 `disconnected`

### 6.3 审批策略
- 默认所有新设备均需人工允许
- Host 支持拒绝连接
- Host 支持对已连接设备执行“断开连接”
- 首期不做永久信任设备白名单

## 7. 通信协议设计

### 7.1 控制通道：WebSocket
用途：
- 握手与审批
- 在线状态同步
- 文本消息发送
- 传输任务创建/取消/失败/完成通知
- 设备断开事件

典型消息类型：
- `hello`
- `connect_request`
- `connect_approved`
- `connect_rejected`
- `device_disconnected`
- `text_message`
- `transfer_offer`
- `transfer_accept`
- `transfer_reject`
- `transfer_progress`
- `transfer_complete`
- `transfer_failed`
- `transfer_cancel`

### 7.2 数据通道：HTTP
用途：
- 文件上传
- 文件下载
- 文件夹清单上传
- 多文件批量流式传输

首期不强制做复杂分片协议，优先实现：
- 单文件流式上传下载
- 多文件按清单逐个传输
- 文件夹通过“相对路径 manifest + 文件流”方式传输

这样比先整体打包 zip 更透明，也更有利于后续做失败定位和单文件重试。

## 8. 文本与文件传输模型

## 8.1 文本
- 文本消息直接通过 `WebSocket` 发送
- 每条文本消息具备唯一 `messageId`
- 支持状态：`sending | sent | failed | received`
- 首期不做已读回执，只做送达/失败状态

### 8.2 文件与多文件
发送流程：
1. 发送方选择文件或多个文件
2. 本地扫描生成 `manifest`
3. 通过 `transfer_offer` 告知接收方待接收内容
4. 接收方选择保存位置
5. 接收方确认后开始 HTTP 传输
6. 控制通道同步任务进度、完成或失败状态

### 8.3 文件夹
文件夹发送流程与多文件一致，但 `manifest` 中每个文件都带相对路径，例如：
- `docs/readme.md`
- `images/a.png`

接收方选择目标保存目录后，系统根据相对路径在本地创建目录结构并写入文件。

### 8.4 保存位置策略
- 手机接收文件/文件夹前，必须选择保存位置
- 电脑接收文件/文件夹前，必须选择保存位置
- 不采用自动保存固定目录

## 9. 多设备并发设计

Host 需要同时管理多个 Client，要求：
- 在线设备列表独立展示
- 每个设备有独立聊天面板和传输任务列表
- 某个设备异常断开不影响其他设备
- 任务状态按 `sessionId` 隔离

桌面端 Host 的 UI 需要提供：
- 待审批设备列表
- 已连接设备列表
- 当前会话切换
- 任务总览和单会话任务视图

## 10. 失败处理与边界条件

首期错误处理策略：
- 不做断点续传
- 不做暂停继续
- 任务失败后允许用户手动重试
- 传输过程中允许任务级取消

需要显式处理的失败场景：
- `pairToken` 无效或过期
- 端口不可用
- 同网段不可达
- 用户拒绝连接
- 用户未选择保存位置
- 文件读取失败
- 目标目录不可写
- 存储空间不足
- 文件名冲突
- 传输过程中网络断开

建议策略：
- 文件名冲突时首期采用自动重命名策略，例如 `name (1).txt`
- 网络断开时将未完成任务直接标记失败
- 失败原因在 UI 中显示可理解文本，而不是原始异常堆栈

## 11. 安全边界

首期安全策略聚焦于局域网内可控连接：
- 通过二维码携带短时有效 `pairToken`
- 必须经由 Host 人工允许
- Host 可随时断开任意设备
- 不开放匿名自动接入
- 不做公网暴露与端到端加密增强能力

说明：
- 局域网环境下仍建议在协议中预留版本号和 token 校验字段
- 首期不实现复杂证书体系或账号登录体系

## 12. 平台实现重点

### 12.1 Desktop / Electron
桌面端需要：
- 启动 Node 层本地服务
- 获取可访问的局域网 IP
- 打开系统文件/目录选择对话框
- 读写本地文件系统
- 渲染二维码
- 支持打包成 `.exe` 与 `.dmg`

### 12.2 Mobile / uni-app Android
移动端需要：
- 扫码能力
- 文件/目录选择能力
- Android 本地存储访问权限处理
- 前台状态下维持连接和传输
- Phase 2 中具备本地服务启动能力

风险说明：
- Android 端“文件夹选择”和“以服务端方式监听局域网端口”都比普通页面逻辑复杂
- 因此首期优先完成移动端作为 Client 的路径闭环

## 13. UI 页面规划

### 13.1 桌面端
- 首页 / Host 控制台
  - 启动/停止服务
  - 显示本机 IP、端口、二维码
  - 待审批设备列表
  - 已连接设备列表
- 会话页
  - 文本消息区域
  - 发送文本输入框
  - 发送文件/多文件/文件夹按钮
  - 传输任务列表

### 13.2 移动端
- 首页
  - 扫码连接
  - 进入接收模式（Phase 2）
- 连接状态页
  - 当前连接中/等待审批/已连接/已断开
- 会话页
  - 文本收发
  - 选择文件
  - 选择文件夹
  - 接收任务与发送任务列表

## 14. 测试策略

测试分层如下：
- `packages/shared`：协议编码、payload 校验、manifest 生成逻辑的单元测试
- `apps/desktop`：连接管理器、审批逻辑、任务状态转换的单元/集成测试
- `apps/mobile`：核心状态管理和协议适配层的单元测试
- 最小端到端验证：桌面 Host + Android Client 完成文本与单文件传输

首期测试重点不是覆盖全部 UI，而是验证：
- 握手与审批状态机正确
- 多会话隔离正确
- 文件夹 manifest 解析正确
- 任务失败状态可追踪

## 16. 实施顺序

推荐实施顺序：
1. 建立 monorepo、共享包、基础脚手架
2. 定义共享协议、类型、消息与状态机
3. 完成桌面端 Host 服务与 UI
4. 完成移动端 Client 扫码、连接、文本消息
5. 完成文件/多文件/文件夹传输闭环
6. 补齐测试、使用说明、打包说明
7. 再扩展移动端 Host 模式，支持手机 ↔ 手机

## 17. 范围确认结论

本设计最终确认以下范围：
- 首期主通信方式为局域网 Wi‑Fi
- 桌面端采用 Electron
- 移动端采用 Vue 3 + Vite + uni-app，目标 Android
- 首期必须支持多手机连接同一 Host
- 连接必须人工允许，并支持手动断开
- 双向文本与文件互传
- 支持多文件与文件夹
- 收到文件时由用户选择保存位置
- 不做断点续传，不做公网穿透，不做蓝牙主链路
